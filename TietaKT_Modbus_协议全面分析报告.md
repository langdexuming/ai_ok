## 二、设备类型说明

本协议用于中国铁塔通信基站专用空调设备的远程监控，属于**动环监控系统**（动力环境监控）的空调监控子系统。

- 符合中国移动动环信号标准化字典表（2024.07.05版）
- 参考协议文档：空调 modbus 协议 V1.0（中性版）
- 支持监控功能：温度/湿度/电压采集、状态/故障告警、远程开关机、温度阈值设置

## 三、通信协议类型

### 3.1 物理层
- **接口**: RS485 半双工
- **波特率**: 9600 bps
- **数据格式**: 8N1 (8 数据位, 无校验, 1 停止位)

### 3.2 数据链路层
- **协议**: Modbus RTU
- **校验**: CRC16 (多项式 0xA001)
- **字节序**: 高位在前 (Big-Endian / HighFirst)

### 3.3 使用的 Modbus 功能码

| 功能码 | 用途 | 说明 |
|---|---|---|
| **0x01** | 读线圈状态 | 读取空调状态/开关量数据 |
| **0x03** | 读保持寄存器 | 读取温度/电压/设定值等模拟量 |
| **0x05** | 写单个线圈 | 空调开机 (0xFF00) / 关机 (0x0000) |
| **0x06** | 写单个寄存器 | 设置制冷温度阈值 |

## 六、请求帧与数据帧格式

### 6.1 Modbus RTU 帧结构

```
[地址码 1B][功能码 1B][数据区 NB][CRC16低 1B][CRC16高 1B]
```

### 6.2 状态数据请求帧 (FC=0x01)

```
发送: 01 01 00 22 00 17 DC 0E
       |  |  |-----|  |-----|  |-----|
       |  |  起始34    数量23   CRC16
       |  功能码0x01
       地址0x01

响应: 01 01 03 [Byte0] [Byte1] [Byte2] [CRC_L] [CRC_H]
       |  |  |   |
       |  |  数据长度3字节
       |  功能码
       地址
```

**Byte0 位定义 (线圈34-41):**
- bit0: 制冷状态 → 通道 10035, 同时输出空调运行模式 → 通道 20001
- bit1: 加热器状态 → 通道 10036
- bit2: 内风机状态 → 通道 10037
- bit3: 外风机状态 → 通道 10038
- bit4: 应急风机状态 → 通道 10039
- bit5-bit6: → 通道 10040, 10041 (代码中有 OUTPUT_VALUE 但未定义通道名)
- bit7: 空调关机告警(取反) → 通道 19042 (bit=1时值为0表示关机, 0时值为1表示运行)

**Byte1, Byte2**: 后续告警位，各 bit 依次映射到 10043-10058 告警通道

### 6.3 温度数据请求帧 (FC=0x03)

```
发送: 01 03 00 00 00 19 84 00
       |  |  |-----|  |-----|  |-----|
       |  |  起始0     数量25   CRC16
       |  功能码0x03
       地址0x01

响应: 01 03 32 [WORD0_H WORD0_L] [WORD1_H WORD1_L] ... [WORD24_H WORD24_L] [CRC_L CRC_H]
       |  |  |   共25个WORD = 50字节数据
       |  |  数据长度50 (0x32)
       |  功能码
       地址
```

每个寄存器 16bit 高位在前读取，若值为 0xFFFF 则替换为 0。

### 6.4 控制帧示例

**空调开机 (FC=0x05):**
```
01 05 00 29 FF 00 [CRC_L] [CRC_H]   (地址41=0x0029, 值0xFF00)
```

**空调关机 (FC=0x05):**
```
01 05 00 29 00 00 [CRC_L] [CRC_H]   (地址41=0x0029, 值0x0000)
```

**设置制冷开启温度为 27.0°C (FC=0x06):**
```
01 06 00 0D 01 0E [CRC_L] [CRC_H]   (地址13=0x000D, 值270=0x010E)
```

## 七、数据解析逻辑详解

### 7.1 状态数据解析 (`UnPackCmdStatusData`)

1. 调用 `CModbusCmd::UnpackCmd()` 验证 CRC16 和帧长度
2. 校验功能码 == 0x01
3. 校验数据字节数 == 3
4. 从 pBuf+3 开始构建 CSafeReadBufferAuto，逐字节读取 3 字节
5. 每字节 8 位逐位提取：
   - `OUTPUT_VALUE(10035 + 8*i + j, byBit)` — 依次输出所有状态位
   - 特殊处理: byte0/bit0 → 同时输出到通道 20001 (空调运行模式)
   - 特殊处理: byte0/bit7 → 取反后输出到通道 19042 (空调运行状态: 1=运行,0=关机)

### 7.2 温度数据解析 (`UnPackCmdRealData`)

1. 验证 CRC16、功能码 0x03、数据长度 50 字节
2. 循环读取 25 个 WORD (高位在前)
3. 对 0xFFFF 值替换为 0 (传感器未接或故障)
4. 换算: 索引 4/5/6/7/24 直接取整数值，其他索引乘以 0.1
5. 输出到 30001~30025 通道

### 7.3 控制命令打包 (`PackCmdSetData`)

1. 检查输入通道数 >= 1
2. 根据控制通道 ID 确定功能码和寄存器地址：
   - 40001/40002: FC=0x05, 地址=41, 开机 0xFF00/关机 0x0000
   - 40014/40015: FC=0x06, 地址=13/14, 浮点值 * 10 转 WORD
3. 调用 `CModbusCmd::PackCmd()` 组帧

### 7.4 控制命令解包 (`UnPackCmdSetData`)

仅检查响应长度 == 8 字节（Modbus 写单个寄存器/线圈的标准回复），不做进一步解析。

IMPLEMENT_COMMAND_STRUCT(name)  // 实现命令分发逻辑

### 8.4 CRC16 计算

采用 Modbus 标准 CRC16 算法, 多项式 0xA001, 初始值 0xFFFF：
- `CModbusCmd::GetCRC16(pBuf, nLen)` — 计算 CRC16
- `CModbusCmd::CheckValid(pBuf, nLen)` — 验证 CRC16
- CRC 在帧中**低字节在前**

解析接收数据:
- 地址=0x01, FC=0x01, 数据长度=3
- Byte0=0x04 (二进制 00000100): bit2=1(内风机运行), 其他=0
- Byte1=0x00, Byte2=0x00: 无告警
- 通道 10037(内风机状态)=1, 其他状态=0
- 通道 19042(空调运行状态): byte0 bit7=0, 取反=1(运行中)
- 通道 20001(空调运行模式): byte0 bit0=0

**温度数据交互:**
```
发送: 01 03 00 00 00 19 84 00
接收: 01 03 32 00 78 00 6E 00 82 00 00 00 00 00 00 00 00 00 00
      00 00 00 00 00 00 00 00 00 00 01 40 01 0E 00 32 00 96 02
      26 00 00 02 F8 00 00 03 20 02 EE 03 84 FF FF 0A DE
```
解析关键寄存器值:
- 寄存器0(柜内温度): 0x0078 = 120 → 12.0°C
- 寄存器1(蒸发器温度): 0x006E = 110 → 11.0°C
- 寄存器2(冷凝器温度): 0x0082 = 130 → 13.0°C
- 寄存器13(制冷开启温度): 0x0140 = 320 → 32.0°C
- 寄存器14(制冷停止温度): 0x010E = 270 → 27.0°C
- 寄存器17(高温告警点): 0x0226 = 550 → 55.0°C
- 寄存器19(冷凝器高温保护): 0x02F8 = 760 → 76.0°C
- 寄存器21(除湿开启湿度): 0x0320 = 800 → 80.0%
- 寄存器22(除湿停止湿度): 0x02EE = 750 → 75.0%
- 寄存器23(湿度告警限值): 0x0384 = 900 → 90.0%
- 寄存器24(湿度校正值): 0xFFFF → 替换为 0 → 直接取值=0

## 十一、代码中可能存在的问题

### 11.1 严重问题

#### (1) 设备地址码硬编码
```cpp
// TietaKT_ModbusCmd.cpp 第9行
m_byAddrCode = 0x01;
```
地址码写死为 0x01，不支持同一总线上多台空调设备。虽然上层可通过 `SetParam(PARAMTYPE_ADDRCODE, ...)` 动态设置，但构造函数中的硬编码可能导致初始化阶段通信时使用错误地址。

#### (2) 状态数据解析越界输出
```cpp
// TietaKT_ModbusCmd.cpp 第50-69行
for (int i=0; i<3; i++) {
    for (int j=0; j<8; j++) {
        OUTPUT_VALUE(10035 + 8*i + j, byBit);  // 输出 10035~10058, 共24个通道
    }
}
```
代码输出了 24 个通道 (10035-10058)，但 `InitChannelData()` 中只定义了 10035-10039 为遥信通道，10042-10058 为告警通道。通道 10040、10041 **未在模板中定义**，但仍会被写入 OutputChannelList，可能导致上层平台出现未知通道数据。

#### (3) 温度数据循环输出未定义通道
```cpp
// TietaKT_ModbusCmd.cpp 第110-134行
for (int i=0; i<25; i++) {
    OUTPUT_VALUE(30001 + i, fValue);  // 输出 30001~30025
}
```
其中 30010~30013 共 4 个通道**在 InitChannelData() 中未定义**，但数据仍会被输出。这些是寄存器地址 9-12 对应的数据，可能是保留寄存器。

#### (4) 控制命令回复解析过于简单
```cpp
// TietaKT_ModbusCmd.cpp 第196-205行
BOOL CTietaKT_ModbusCmd::UnPackCmdSetData(BYTE *pBuf, int nSize, ...)
{
    if(nSize != 8) return FALSE;
    return TRUE;
}
```
仅校验长度为 8，**不验证 CRC16、不验证功能码、不验证返回的寄存器地址和值是否与发送一致**。无法检测设备是否正确执行了控制命令。

### 11.2 潜在问题

#### (5) ControlChannelMap 内存泄漏
`ProtocolConfig.cpp` 中 `MapControlChannel()` 使用 `new` 分配 ControlChannelMap 对象存入数组，但析构函数 `~CProtocolConfig()` 中**未释放这些对象**，只释放了 `m_ChannelParamMap` (在 PARSE_AUTO 条件下)。

#### (6) CModbusCmd 析构时的静态成员清理
```cpp
// ModbusCmd.cpp 第13-21行
CModbusCmd::~CModbusCmd(void)
{
    for(int i=0; i<m_CmdParam.size(); i++)
        delete m_CmdParam[i];
    m_CmdParam.clear();
}
```
`m_CmdParam` 是 **static** 成员，在任意一个 CModbusCmd 实例析构时就会清空，这会影响其他仍在使用的实例。不过本协议未使用 PARSE_AUTO 模式，不会触发此问题。

#### (7) StdCmdBase::GetValueWithParam 中 DATATYPE_INT case 缺少 break
```cpp
// StdCmdBase.cpp 第509-527行
case DATATYPE_INT:
    {
        // ... 处理逻辑
    }
// 注意: 这里没有 break; 会 fall-through 到 DATATYPE_SHORT
case DATATYPE_SHORT:
```
`DATATYPE_INT` 处理完后缺少 `break`，会 fall-through 到 `DATATYPE_SHORT` 的处理逻辑，导致 INT 类型数据被错误地再次解析为 SHORT。

#### (8) 版本修改记录不完整
代码中设置版本为 `4.0.1.2`，但版本修改记录文件只记录了 `4.0.1.1`，`4.0.1.2` 版本的改动内容未记录。

#### (9) 缓冲区大小未校验
`PackCmd` 系列函数中，传入的 `pBuf` 缓冲区大小未做检查。例如 `PackCmd()` 假设缓冲区至少有 8 字节，`PackCmd10()` 假设至少有 `9+iDataLen` 字节，但均未验证 `nSize` 参数是否足够。

#### (10) 温度设置值的负数处理
```cpp
// TietaKT_ModbusCmd.cpp 第172-177行
wValue = (WORD)(fValue*10);
```
当设置温度为负值时（例如 -5.0°C），`fValue*10 = -50`，强制转换为 `WORD`(unsigned short) 会得到错误的大数 (65486)，不会正确传输负温度值。

#### (11) 轮询调度中的死循环风险
```cpp
// ProtocolConfig.cpp GetNextCircleCmdID 带频率版本
while(1) {
    dwResult = GetNextCircleCmdID(dwID);
    // ...
    dwID = dwResult;
}
```
如果所有命令的频率条件都不满足且不会返回 0，理论上可能死循环。实际中由于 `GetNextCircleCmdID` 最终会返回 0，风险较低但代码缺乏保护。

### 11.3 代码风格/可维护性问题

- 大量使用魔术数字（线圈起始地址 34、寄存器数量 23/25、控制寄存器 41 等），缺乏常量定义
- 协议调试举例文件中 SEND 字段为空，缺少完整的收发样例
- 状态位到告警通道的映射关系隐含在循环逻辑中（`10035 + 8*i + j`），不够直观
- 温度数据的换算系数（0.1）散落在 switch-case 中，不利于维护
