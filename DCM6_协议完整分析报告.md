## 二、协议帧格式

采用 ASCII 编码的电总协议帧格式：

```
SOI   VER   ADR   CID1   CID2   LENGTH   INFO   CHKSUM   EOI
起始位 版本号 设备地址 标识码   RTN     长度     数据     校验    结束码
 1     1      1      1      1      2       n       2       1
```

| 字段 | 字节数 | 说明 |
|------|--------|------|
| SOI | 1 | 起始位: `0x7E` (波浪号'~') 或 `0x28` (左括号'(') |
| VER | 1 | 协议版本号, 默认 0x20 |
| ADR | 1 | 设备地址, 默认 0x01 |
| CID1 | 1 | 命令标识码1, 固定 0x60 (空调设备) |
| CID2 | 1 | 命令标识码2 / 返回码RTN |
| LENGTH | 2 | 数据域长度(含4位校验) |
| INFO | n | 数据域 |
| CHKSUM | 2 | 校验和 (取反+1) |
| EOI | 1 | 结束码: `0x0D` (回车CR) 或 `0x29` (右括号')') |

**数据传输特点**: 所有字段以 ASCII 十六进制字符串传输, 每个字节用2个ASCII字符表示。

**LENGTH编码**: 采用4位校验 + 12位数据长度, 校验为低12位各半字节之和取反+1放在最高4位。

**CHKSUM**: 从VER到INFO所有字节的ASCII值累加, 取反+1。

## 五、数据解析逻辑详解

### 5.1 模拟量数据解析 (UnPackCmdGetAnalogData)

1. 验证CID1=0x60, CID2=0x00(正常返回)
2. 支持数据长度 37/38/40 字节三种格式
3. 当长度为38或40时, 跳过1字节并设置 `m_bFlag=TRUE` (版本兼容标志)
4. 读取12个WORD作为通道1001-1012:
   - 当值为0x2020(空格)时设为0
   - 第7、8个值(i=6,7 即送风温度、回风温度)按有符号short处理
5. 读取1字节表示后续扩展字段数量(最大7个)
6. 逐个读取扩展字段写入1013+偏移:
   - 第1、2、4个(i=0,1,3)按有符号short处理

### 5.2 开关状态解析 (UnPackCmdGetOnoffData)

1. 使用 `UnPackCmdASCIIEX` 解包(带校验验证)
2. 跳过1字节后读取运行状态字节
3. **版本兼容**: 如果 `m_bFlag=TRUE` (由模拟量命令设置), 对运行状态值取反(0变1, 1变0)
4. 据运行状态值生成空调关机告警(4001)
5. 解析运行模式并转换为集采标准值
6. 读取6个状态字节: 内风机/四通阀/压缩机/外风机/摆风/电加热
7. 内风机状态同时写入风速设定(3103)
8. 压缩机状态取反写入压缩机停机告警(4004)

### 5.3 告警数据解析 (UnPackCmdGetmAlarmData)

1. 最小数据长度38字节
2. 跳过1+8=9字节
3. 读取3字节: 过滤网保护/压缩机保护/风机保护 (1309-1311)
4. 跳过1字节(自定义数量值0x25)
5. 读取3字节: 高压保护/低压保护/排气温度保护 (1313-1315)
6. 跳过12字节
7. 读取9字节: 室外风机保护 ~ 外机被盗告警 (1327-1335)
8. 告警判定: 0x00或0x20为正常, 其他值为告警

### 5.4 空调参数解析 (UnPackCmdGetmParmData)

1. 最小数据长度33字节
2. 读取第1个WORD为温度值1(fTemp1)
3. 跳过5个WORD后读取温度值2(fTemp2)
4. 读取1字节判断空调类型:
   - **byValue=9或0**: 铁塔普通空调, 使用fTemp2作为温度设定值
   - **byValue=12**: 铁塔机柜空调, 使用fTemp1作为温度设定值, 设置 `m_bFlagTemp=TRUE`
5. 读取运行模式WORD, 写入6000/6001(原始值)和3102(集采映射值)

### 5.5 电量数据解析 (UnPackCmdGetEnergyData)

1. CID2=0x04时表示不支持, 直接返回TRUE
2. 最小数据长度6字节
3. 读取4字节DWORD乘以0.01 -> 累计总电量(2001)
4. 读取2字节WORD乘以0.01 -> 今日电量(2002)

## 九、串口日志分析

### 9.1 通信参数确认

- 串口: COM, 9600bps, 8数据位, 1停止位, 无校验
- 轮询方式: [POLL] 循环轮询, [CTRL] 控制命令

### 9.2 轮询周期

每轮完整采集包含7个命令, 周期约9秒:
1. 获取告警数据 (CID2=0x44) -> 94字节响应
2. 获取空调参数 (CID2=0x47) -> 84字节响应
3. 获取空调电量 (CID2=0x81) -> 18字节响应 (CID2=0x04不支持)
4. 获取版本号 (CID2=0x4F) -> 18字节响应
5. 获取地址 (CID2=0x50) -> 18字节响应
6. 获取模拟量 (CID2=0x42) -> 94字节响应
7. 获取开关状态 (CID2=0x43) -> 42字节响应 (解析后 timeout)

### 9.4 典型问题 -- 开关状态解析timeout

在日志中反复出现: 获取开关状态(0x43)命令发送后收到42字节响应, 但随后出现 `contact with dev[1] time out`。这表明开关状态响应数据中有0x00字节(空字节), 导致解析可能异常或后续命令发送延迟。

## 十、代码中可能存在的问题

### 10.1 严重问题

1. **switch case fall-through (DCM6Cmd.cpp:468-472)**:
   在运行模式解析中, `case 1:` 设置 `byCold=1` 后没有break, 会fall-through到 `case 2:` 和 `case 4:`. 当模式为1(制冷)时, byStatus会被设为1(正确), 但当模式为4(制热)时, byHot没有被设置为1. 制热状态永远为0.

2. **ChkLenCode校验被注释掉 (DCM6Cmd.cpp:32-33)**:
   v4.0.1.5中 `ChkLenCode` 和 `ChkLenCodeEX` 的长度校验失败时返回FALSE被注释为 `//return FALSE`, 这意味着长度校验形同虚设, 可能接受格式错误的数据包.

3. **AsciiToHexEX中的逻辑错误 (v4.0.1.5 DCM6Cmd.cpp:1072)**:
   `if ((chAsciiH < 0x30) || (chAsciiL < 0x30))` 这种处理方式过于宽泛, 会将所有小于0x30的字符视为0x00, 可能掩盖数据错误.

4. **DCM8版本AsciiToHexEX bug (适用DCM8 DCM6Cmd.cpp:950)**:
   `if ((chAsciiH ==0x20) && (chAsciiH ==0x20))` -- 判断条件检查了两次chAsciiH, 应该是 `chAsciiH ==0x20 && chAsciiL ==0x20`. 同样 `if ((chAsciiH ==0x00) && (chAsciiH ==0x00))` 也检查了两次chAsciiH.

### 10.2 中等问题

5. **硬编码的地址和版本号**: `m_byAddrCode = 0x01` 和 `m_byVer = 0x20` 写死在构造函数中, 不支持配置文件方式修改.

6. **固定大小的栈缓冲区**: 多处使用 `BYTE arrData[256]` 和 `BYTE arrData[512]`, 没有动态分配. 如果设备返回超大数据包, 虽有长度检查但仍存在风险.

7. **电量命令的CID2检查不完整 (DCM6Cmd.cpp:707-713)**: 只检查了 `CID2==0x04` 作为不支持直接返回, 但没有检查 `CID2==0x00` 作为正常返回, 其他CID2值也会通过.

8. **告警通道编号不连续**: 1312, 1316, 1326, 1330, 1331, 1334等编号被跳过, 说明这些告警位在协议中存在但程序未实现.

### 10.3 轻微问题

9. **strcpy使用不安全**: `strcpy(m_ProtocolInfo.acName, "DCM6")` 没有使用strncpy, 存在缓冲区溢出风险(虽然实际"DCM6"很短).

10. **m_bFlag全局状态耦合**: 版本兼容标志 `m_bFlag` 在模拟量解析中设置, 在开关状态解析中使用. 如果命令执行顺序变化或模拟量解析失败, 开关状态解析的行为会不正确.

11. **m_bFlagTemp全局状态**: 类似 `m_bFlag`, `m_bFlagTemp` 在参数解析中设置, 在控制命令打包中使用, 存在执行顺序依赖.

12. **告警数据固定跳过字节数**: `safeBuf.JumpBack(8)` 和 `safeBuf.JumpBack(12)` 是硬编码的偏移量, 如果设备固件更新改变了数据布局, 解析会出错.

13. **模拟量解析的数据长度判断 (v4.0.1.5)**: `if (iDataSize != 37 && iDataSize != 38 && iDataSize != 40)` 只接受3种固定长度, 扩展性差.

14. **命令ID 1和2的描述重复**: 两个命令都叫"获取地址版本号", 实际功能不同(一个获取版本, 一个获取地址).
