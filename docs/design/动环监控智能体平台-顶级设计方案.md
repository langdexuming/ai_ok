## 1. 项目概述

### 1.1 背景

动力环境（动环）监控系统是通信基站和数据中心运维的核心基础设施，负责对电力、空调、环境、安防等设备进行7×24小时实时监控。随着站点规模扩大和设备种类增多，传统人工巡检和被动告警模式面临以下挑战：

- **告警量爆炸**：日均告警量达百万级，大量为无效告警、重复告警，运维人员疲于应对
- **异常发现滞后**：设备信号缺失、监控盲区等问题难以主动发现，往往等到故障发生才暴露
- **诊断效率低**：异常根因分析依赖资深人员经验，新人上手难，处理周期长
- **闭环管理缺失**：异常预警从发现到处置缺少全流程跟踪，问题容易遗漏
- **应急响应迟缓**：自然灾害、大面积停电等应急场景下，无法快速掌握全局态势

### 1.2 项目定位

本项目构建 **动环监控智能体平台**，以 AI Agent（智能体）为核心引擎，将传统被动式监控升级为 **主动感知 → 智能诊断 → 自动跟踪 → 闭环处置** 的智能运维体系。

### 1.3 核心理念

```
┌─────────────────────────────────────────────────────────┐
│                    智能体核心理念                          │
│                                                         │
│   感知层：全量数据接入，主动发现异常                        │
│      ↓                                                  │
│   认知层：知识图谱 + AI模型，理解异常本质                   │
│      ↓                                                  │
│   决策层：关联分析定位根因，生成处置建议                     │
│      ↓                                                  │
│   执行层：工单派发、跟踪闭环、应急联动                      │
│      ↓                                                  │
│   进化层：处置经验反馈，知识持续沉淀                        │
└─────────────────────────────────────────────────────────┘
```

## 2. 业务目标与核心价值

### 2.1 量化目标

| 目标指标 | 当前状态 | 目标值 | 提升幅度 |
|---------|---------|-------|---------|
| 动环监控覆盖率 | ~85% | ≥95% | 同比提升10%+ |
| 动环可用度 | ~80% | ≥90% | 同比提升10%+ |
| 非法告警屏蔽操作 | 基数较大 | 减少80%+ | 大幅下降 |
| 异常诊断平均耗时 | 2-4小时 | ≤30分钟 | 提升4-8倍 |
| 监控设备缺失率 | ~5% | ≤1% | 趋近零缺失 |
| 应急响应启动时间 | 30-60分钟 | ≤5分钟 | 提升6-12倍 |
| 异常工单闭环率 | ~70% | ≥95% | 大幅提升 |

### 2.2 核心价值

- **全面感知**：8大智能体模块全方位覆盖动环监控的每个环节
- **智能诊断**：AI驱动的根因分析替代人工经验判断，降低技能门槛
- **闭环管理**：异常从发现到处置全流程跟踪，杜绝遗漏
- **主动预防**：从被动告警响应转向主动异常预测和风险预警
- **全局可视**：统一大屏展示，一览全局态势

### 4.2 Agent 核心属性

```
Agent {
    id          : 唯一标识
    name        : 智能体名称
    type        : 类型（监测类 / 诊断类 / 分析类 / 响应类）
    schedule    : 调度策略（实时 / 定时 / 事件触发）
    perception  : 感知配置（数据源、订阅主题、过滤规则）
    reasoning   : 推理配置（模型ID、知识域、规则集）
    action      : 行动配置（输出通道、通知策略、工单模板）
    memory      : 记忆配置（上下文窗口、知识库引用）
    status      : 运行状态（运行中 / 暂停 / 异常）
    metrics     : 运行指标（处理量、准确率、响应时间）
}
```

## 5. 功能模块详细设计

### 5.1 系统异常监测

> **Agent名称**：异常监测智能体（Anomaly Detection Agent）
> **调度模式**：实时流式处理
> **核心定位**：全面监测异常情况，统一预警展示，一览全局

#### 5.1.1 监测维度

| 监测项 | 说明 | 检测方法 | 触发条件 |
|--------|------|---------|---------|
| **监控设备缺失** | 应有监控但未接入或离线的设备 | 综资设备清单与动环在线设备比对 | 设备离线超过阈值时间 |
| **设备信号值异常** | 设备上报的信号值超出合理范围 | 基于设备类型的信号值范围校验 + 统计异常检测 | 超出静态阈值或动态基线 |
| **设备信号缺失** | 设备在线但特定信号长时间未上报 | 信号上报周期监控，心跳超时检测 | 信号缺失时间超过N个采集周期 |
| **设备信号异常波动** | 信号值短时间内剧烈变化 | 滑动窗口标准差、变化率分析 | 波动幅度超过历史基线的N倍 |
| **告警频繁跳变** | 同一告警短时间内反复产生和恢复 | 滑动窗口内告警状态翻转次数统计 | 单位时间内跳变次数超过阈值 |
| **油机发电过程** | 监控油机启动、运行、停止全过程 | 油机电流/电压/频率信号状态机 | 状态转换不符合预期序列 |

#### 5.1.2 检测算法体系

```
┌─────────────────────────────────────────────────────┐
│                 异常检测算法分层                       │
│                                                     │
│  第一层：规则检测（Rule-Based）                       │
│  · 静态阈值检测（上下限）                             │
│  · 信号有效范围校验                                   │
│  · 心跳超时检测                                      │
│  · 状态翻转频率检测                                   │
│                                                     │
│  第二层：统计检测（Statistical）                      │
│  · 3σ原则（正态分布异常检测）                          │
│  · Z-Score 标准化异常评分                             │
│  · IQR 四分位距异常检测                               │
│  · 滑动窗口趋势分析                                   │
│                                                     │
│  第三层：AI检测（Model-Based）                        │
│  · 时序异常检测模型（Isolation Forest / LSTM-AE）      │
│  · 多信号关联异常检测                                  │
│  · 设备行为画像偏离检测                                │
│  · 季节性分解异常检测（STL）                           │
│                                                     │
│  融合层：多层检测结果加权融合，输出综合异常评分           │
└─────────────────────────────────────────────────────┘
```

#### 5.1.4 统一预警展示

预警中心按以下维度统一展示所有异常：

- **地域维度**：省 → 地市 → 区县 → 站点 → 设备
- **严重等级**：紧急（红）→ 重要（橙）→ 一般（黄）→ 提示（蓝）
- **异常类型**：设备缺失 / 信号异常 / 信号缺失 / 信号波动 / 告警跳变 / 油机异常
- **时间趋势**：实时 → 小时 → 天 → 周 → 月

### 5.3 异常跟踪

> **Agent名称**：异常跟踪智能体（Anomaly Tracking Agent）
> **调度模式**：事件触发 + 定时巡检
> **核心定位**：异常预警全流程跟踪处理，形成闭环

#### 5.3.1 异常生命周期

```
  ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
  │  发现   │──→│  确认   │──→│  派发   │──→│  处理   │──→│  验证   │
  │Detected│   │Confirmed│  │Assigned│   │Handling│   │Verified│
  └────────┘   └────────┘   └────────┘   └────────┘   └────────┘
       │            │            │            │            │
       ↓            ↓            ↓            ↓            ↓
  异常监测     AI诊断确认    工单自动派发   运维人员处理   系统自动验证
  Agent发现    或人工确认    到责任人      线上/线下      信号恢复正常
       │                                                   │
       │         ┌────────┐                                │
       │         │  关闭   │←───────────────────────────────┘
       │         │ Closed  │
       │         └────────┘
       │              │
       └──超时升级────→│
                      ↓
                 ┌────────┐
                 │  升级   │
                 │Escalated│
                 └────────┘
```

#### 5.3.2 跟踪规则引擎

| 规则类型 | 触发条件 | 执行动作 |
|---------|---------|---------|
| **自动确认** | AI诊断置信度 ≥ 90% | 自动确认异常，跳过人工确认 |
| **自动派发** | 异常确认后 | 根据站点归属、异常类型自动匹配责任人 |
| **超时提醒** | 处理时间超过SLA的50% | 发送提醒通知给处理人 |
| **超时升级** | 处理时间超过SLA | 自动升级到上级主管 |
| **自动验证** | 处理完成后 | 持续监测对应信号是否恢复正常 |
| **自动关闭** | 验证通过且稳定运行N小时 | 自动关闭异常工单 |
| **重复检测** | 同一设备同类异常再次发生 | 标记为重复异常，关联历史记录 |

#### 5.3.3 SLA 定义

| 异常等级 | 响应时限 | 处理时限 | 升级策略 |
|---------|---------|---------|---------|
| 紧急（P0） | 15分钟 | 2小时 | 超时后每30分钟逐级升级 |
| 重要（P1） | 30分钟 | 4小时 | 超时后每1小时逐级升级 |
| 一般（P2） | 2小时 | 24小时 | 超时后每4小时提醒一次 |
| 提示（P3） | 4小时 | 72小时 | 超时后每天提醒一次 |

#### 5.3.4 闭环统计指标

- 异常发现到确认的平均耗时
- 异常确认到派发的平均耗时
- 异常处理的平均耗时
- 异常一次处理成功率
- 异常工单闭环率
- 超时升级率
- 重复异常率

### 5.5 监控缺失保障

> **Agent名称**：监控缺失智能体（Monitor Gap Agent）
> **调度模式**：定时执行（每日/每小时）
> **核心定位**：实时分析设备监控缺失，保障设备监控零缺失

```

#### 5.5.2 缺失分类

| 缺失级别 | 说明 | 检测方式 |
|---------|------|---------|
| **站点缺失** | 综资有站点记录但动环无该站点 | 站点编码比对 |
| **设备缺失** | 站点已接入但部分设备未监控 | 设备类型与数量比对 |
| **信号缺失** | 设备已接入但部分信号未配置或未上报 | 信号配置完整性校验 |
| **数据缺失** | 信号已配置但数据长时间未更新 | 最近数据时间戳检查 |

#### 5.5.3 整改跟踪

每条缺失记录生成整改工单，跟踪整改进度直至闭环：

- 自动识别缺失原因（未接入/设备故障/网络中断/配置遗漏）
- 自动派发给对应地市/区县负责人
- 定期生成缺失率报表（按地市、按设备类型）
- 缺失率趋势追踪，目标驱动至零缺失

│  │ 数据完整性   │  25%    信号数据缺失率，数据断点统计         │

│  │ 信号值范围   │  20%    信号值在合理范围内的比例              │
│  ├─────────────┤                                           │
│  │ 告警有效性   │  20%    有效告警占比，排除误告警/频繁跳变     │
│  ├─────────────┤                                           │
│  │ 在线时长     │  20%    设备在线时长占比(月度在线率)          │
│  ├─────────────┤                                           │
│  │ 数据存储率   │  15%    数据成功存储占应存储总量的比例        │
│  └─────────────┘                                           │
│                                                             │
│  综合可用度 = Σ(维度评分 × 权重)                              │
└─────────────────────────────────────────────────────────────┘
```

#### 5.7.3 可用度分析报告

- **维度拆解**：各评估维度的得分分布
- **短板分析**：可用度最低的站点/设备/地市 Top N

- **趋势分析**：月度可用度变化趋势，同比环比

### 5.8 告警屏蔽分析

> **Agent名称**：告警屏蔽分析智能体（Alarm Shielding Analysis Agent）
> **调度模式**：定时执行（每日）+ 实时监控
> **核心定位**：核查告警屏蔽合规性，预期非法屏蔽操作减少80%+

## 10. 非功能性需求

### 10.1 性能要求

| 指标 | 要求 |
|------|------|
| 异常检测延迟 | 从信号数据到达到异常事件产生 ≤ 5秒 |
| AI诊断延迟 | 从异常触发到诊断报告生成 ≤ 30秒 |
| 大屏刷新频率 | 实时数据 ≤ 5秒刷新，统计数据 ≤ 1分钟刷新 |
| API响应时间 | P99 ≤ 500ms（查询类），P99 ≤ 2s（分析类） |
| 数据吞吐量 | 支持每秒处理 ≥ 10万条信号数据 |
| 并发用户数 | 支持 ≥ 500并发用户 |

### 10.2 可用性要求

| 指标 | 要求 |
|------|------|
| 系统可用性 | ≥ 99.9%（年停机时间 ≤ 8.76小时） |
| 数据持久性 | ≥ 99.99% |
| 故障恢复时间（RTO） | ≤ 30分钟 |
| 数据恢复点（RPO） | ≤ 5分钟 |

### 10.3 安全要求

- 身份认证：统一认证（SSO）+ OAuth2.0
- 权限控制：RBAC 角色权限体系，支持数据权限（地域隔离）
- 数据加密：传输加密（TLS 1.3）+ 敏感数据存储加密
- 审计日志：所有操作留痕，屏蔽操作完整记录
- 接口安全：API限流、防重放、参数校验
