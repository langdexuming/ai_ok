## 1. 项目概述

### 1.1 背景与痛点

动力环境（动环）监控系统负责对通信基站和数据中心的电力、空调、环境、安防等设备进行7×24小时实时监控。当前面临的核心痛点：

| 痛点 | 现状描述 |
|------|---------|
| 告警量爆炸 | 日均百万级告警，大量无效/重复，运维人员疲于应对 |
| 异常发现滞后 | 信号缺失、监控盲区等问题被动暴露 |
| 诊断效率低 | 根因分析依赖资深经验，处理周期2-4小时 |
| 闭环缺失 | 异常从发现到处置无全流程跟踪，容易遗漏 |
| 应急迟缓 | 灾害/大面积停电时无法快速掌握全局态势 |
| 数据治理弱 | 覆盖率、可用度统计被动，屏蔽操作缺乏管控 |

### 1.2 项目定位

构建 **"动环运维大脑"智能体平台**，将传统被动监控升级为：

```
主动感知 → 智能诊断 → 自动跟踪 → 闭环处置 → 知识沉淀
```

### 1.3 传统平台 vs 智能体平台

| 维度 | 传统动环平台 | 智能体平台（目标） |
|------|------------|------------------|
| 告警处理 | 靠人盯、靠经验 | AI预诊断，处理效率提升50%+ |
| 异常发现 | 被动接收告警 | 主动识别异常，包括"信号伪装" |
| 监控覆盖 | 静态统计，更新慢 | 动态对账，实现100%覆盖保障 |
| 故障定位 | 单点孤立告警 | 知识图谱关联，多告警联合定位根因 |
| 数据治理 | 业务与数据脱节 | 持续审计，可用度提升10%以上 |
| 应急指挥 | 手动调数据、找视频 | 一键响应，决策时间从分钟级降至秒级 |
| 屏蔽管控 | 无审计，随意屏蔽 | 合规审计+策略推荐，非法屏蔽减少80% |

### 4.1 四角色 × 八Agent

采用"4角色分组 + 8专职Agent"的双层模型 —— 对外用4角色说清全局，对内按8个Agent精细化开发部署。

```
┌──────────────────────────────────────────────────────────────┐
│                 动环运维大脑 — 智能体全景                        │
│                                                              │
│  ┌─ 感知智能体 The Observer ──────────────────────────────┐   │
│  │  异常监测Agent    实时识别7大类异常（含信号伪装检测）      │   │
│  │  监控缺失Agent    正向缺失+反向僵尸清理，保障零缺失       │   │
│  └────────────────────────────────────────────────────────┘   │
│                           ↓ 异常事件                          │
│  ┌─ 诊断智能体 The Analyst ──────────────────────────────┐   │

│  └────────────────────────────────────────────────────────┘   │
│                           ↓ 诊断结论                          │
│  ┌─ 调度智能体 The Responder ────────────────────────────┐   │
│  │  异常跟踪Agent    全生命周期闭环，SLA驱动，NLP过程监控   │   │
│  │  应急响应Agent    战时指挥模式，GIS+视频+资源调度联动     │   │

│  ┌─ 治理智能体 The Auditor ──────────────────────────────┐   │

### 4.2 Agent 统一模型

每个Agent遵循统一的"感知-推理-行动-记忆"生命周期：

```
Agent {
    id          : 唯一标识
    name        : 智能体名称
    role        : 所属角色（Observer / Analyst / Responder / Auditor）
    schedule    : 调度策略（实时 / 定时 / 事件触发）
    perception  : 感知配置（数据源、订阅主题、过滤规则）
    reasoning   : 推理配置（模型ID、知识域、规则集）
    action      : 行动配置（输出通道、通知策略、工单模板）
    memory      : 记忆配置（短期上下文 + 长期知识库）
    status      : 运行状态（运行中 / 暂停 / 待命 / 异常）
    metrics     : 运行指标（处理量、准确率、响应时间）
}
```

## 5. 功能模块详细设计

### 5.1 系统异常监测（Observer - 异常监测Agent）

> **调度模式**：实时流式
> **核心定位**：全面监测异常情况，包括隐蔽的"信号伪装"，统一预警展示

#### 5.1.1 七大监测维度

| # | 监测项 | 说明 | 检测方法 | 触发条件 |
|---|--------|------|---------|---------|
| 1 | **监控设备缺失** | 应有监控但未接入或离线 | 综资清单与动环在线设备比对 | 离线超过阈值时间 |
| 2 | **设备信号值异常** | 信号值超出合理范围 | 信号值范围校验 + 统计异常检测 | 超出静态阈值或动态基线 |
| 3 | **设备信号缺失** | 在线但信号长时间未上报 | 心跳超时检测 | 缺失超N个采集周期 |
| 4 | **设备信号异常波动** | 信号短时间剧烈变化 | 滑动窗口标准差+变化率分析 | 波动超历史基线N倍 |
| 5 | **告警频繁跳变** | 同一告警反复产消 | 状态翻转次数统计 | 单位时间跳变超阈值 |
| 6 | **油机发电过程** | 油机启动/运行/停止监控 | 电流/电压/频率状态机 | 状态转换不符预期 |
| 7 | **信号伪装检测** | 设备"假正常"状态 | 冻结检测+临界值抖动+逻辑一致性 | 见下方细则 |

**信号伪装检测细则**（识别设备"假正常"）：

| 伪装类型 | 现象 | 检测方法 | 典型场景 |
|---------|------|---------|---------|
| **信号冻结** | 信号值长时间呈完美直线（方差≈0） | 滑动窗口方差检测 | 采集模块假死，上报缓存旧值 |
| **临界值抖动** | 信号规律性地在阈值上下小幅跳动 | 阈值±δ范围占比+翻转频率分析 | 设备处于故障边缘 |
| **状态矛盾** | 多信号间存在逻辑矛盾 | 多信号逻辑一致性校验 | 油机ON但发电电流=0 |

#### 5.1.2 三层检测算法体系

```
第一层：规则检测（Rule-Based）
  · 静态阈值（上下限）  · 信号有效范围校验
  · 心跳超时检测        · 状态翻转频率检测

第二层：统计检测（Statistical）
  · 3σ原则  · Z-Score  · IQR四分位距  · 滑动窗口趋势

第三层：AI检测（Model-Based）
  · 时序异常检测（Isolation Forest / LSTM-AE）
  · 多信号关联异常检测
  · 设备行为画像偏离检测
  · 信号冻结检测（方差趋零）
  · 多信号逻辑一致性校验

融合层：多层结果加权融合，输出综合异常评分
```

#### 5.1.3 数据流

```
设备信号流(Kafka) → 数据清洗 → 规则检测 → 统计检测 → AI检测 → 预警事件(Kafka)
告警事件流(Kafka) → 告警解析 → 跳变分析 ──────────────────────→ 预警中心
设备台账(综资)   → 状态比对 → 缺失/伪装检测 ─────────────────→ 预警中心
```

#### 5.1.4 统一预警展示

按四个维度统一展示所有异常：
- **地域**：省 → 地市 → 区县 → 站点 → 设备
- **等级**：紧急（红）→ 重要（橙）→ 一般（黄）→ 提示（蓝）
- **类型**：设备缺失/信号异常/信号缺失/信号波动/告警跳变/油机异常/信号伪装
- **时间**：实时 → 小时 → 天 → 周 → 月

### 5.2 异常预警AI诊断（Analyst - AI诊断Agent）

> **调度模式**：事件触发（接收异常监测Agent输出）
> **核心定位**：动态知识图谱关联分析 + LLM辅助，定位根因，生成诊断报告

#### 5.2.1 诊断流程

```
异常事件 → 上下文补全 → 知识检索 → 根因推理 → 诊断报告
             │              │            │
        · 设备属性     · 历史案例    · 故障树分析
        · 站点信息     · 故障模式    · 关联事件分析
        · 关联告警     · 设备拓扑    · 概率推理
        · 动态状态     · 运维知识    · LLM辅助推理
```

#### 5.2.2 动态知识图谱

知识图谱包含**静态拓扑**和**实时运行状态**两层：

| 层级 | 内容 | 更新频率 |
|------|------|---------|
| **静态层** | 设备类型/关联拓扑/供电链路/制冷链路/信号映射 | 天级（跟随台账变更） |
| **动态层** | 设备运行模式(市电/油机/蓄电池)/负载率/环境实时值/告警活跃状态 | 秒级（Redis同步） |

实体关系：

| 实体 | 关系 | 目标 | 说明 |
|------|------|------|------|
| 站点 | 包含 | 设备 | 站点下属设备 |
| 设备 | 产生 | 信号 | 设备采集的信号 |
| 设备 | 关联 | 设备 | 设备间物理/逻辑关联（供电/制冷链路） |
| 信号 | 触发 | 告警 | 信号异常触发告警 |
| 告警 | 对应 | 故障模式 | 告警匹配故障模式 |
| 故障模式 | 指向 | 根因 | 故障模式的根因 |
| 根因 | 关联 | 处理方法 | 对应处置方案 |
| 处理方法 | 参考 | 历史案例 | 历史处理记录 |

#### 5.2.3 关联推理案例

**案例一：油机带载能力不足导致空调停机**

```
现象：某站点空调高温告警（42°C）+ 整流柜输出电流激增

推理链路：
  1. 查询动态上下文 → 该站点当前"油机发电"模式
  2. 图谱关联 → 油机 → 供电链路 → 整流柜 + 空调
  3. 数据校验 → 油机输出198V（额定220V），带载率95%
  4. 根因 → 油机带载不足 → 电压偏低 → 空调欠压保护停机 → 温升

处理建议：
  紧急：检查油机输出电压和带载能力
  短期：减少非核心负载或调度第二台油机并联
  长期：评估站点负荷与油机容量匹配度
```

**案例二：信号冻结掩盖真实故障**

```
现象：A站点温度信号连续6小时=25.0°C（方差=0），相邻B站点36°C

推理链路：
  1. 信号伪装检测 → 方差=0，判定信号冻结
  2. 图谱关联 → A站点空调状态信号也为固定值
  3. 数据比对 → 相邻站点差异过大
  4. 根因 → 温度采集模块假死 → 上报缓存旧值 → 真实温度未知

处理建议：
  紧急：安排人员现场确认实际温度
  短期：更换温度采集模块
  防范：将信号冻结检测纳入常态化监控
```

### 5.3 异常跟踪（Responder - 异常跟踪Agent）

> **调度模式**：事件触发 + 定时巡检

#### 5.3.1 异常生命周期

```
发现(Detected) → 确认(Confirmed) → 派发(Assigned) → 处理(Handling) → 验证(Verified) → 关闭(Closed)
     │                                                                      │
     └─── 超时升级(Escalated) ←──────────────────────────────────────────────┘

#### 5.3.2 跟踪规则

| 规则 | 触发条件 | 执行动作 |
|------|---------|---------|
| 自动确认 | AI置信度 ≥ 90% | 跳过人工确认 |
| 自动派发 | 异常确认后 | 按站点归属+异常类型匹配责任人 |
| 超时提醒 | 处理时间超SLA 50% | 通知处理人 |
| 超时升级 | 处理时间超SLA | 逐级升级上级主管 |
| 自动验证 | 处理完成后 | 24小时观察期，监测信号恢复且无重发/跳变 |
| 自动关闭 | 24小时观察期通过 | 关闭工单 |
| 重复检测 | 同设备同类异常再发 | 标记重复，关联历史 |
| **NLP过程监控** | 工单回复更新时 | NLP分析处理描述与故障的匹配度，识别敷衍回复 |

### 5.5 监控缺失保障（Observer - 监控缺失Agent）

> **调度模式**：定时（每日/每小时）
> **核心定位**：正向缺失检测 + 反向僵尸清理，保障零缺失

#### 5.5.1 双向检测

| 方向 | 检测内容 | 说明 |
|------|---------|------|
| **正向（应有未接入）** | 综资有记录但动环无该站点/设备 | 传统缺失检测 |
| **反向（已拆未下线）** | 综资已拆除但动环仍在监控 | 清理僵尸记录，避免虚假覆盖率 |

#### 5.5.2 缺失分级

| 级别 | 说明 | 检测方式 |
|------|------|---------|
| 站点缺失 | 综资有但动环无 | 站点编码比对 |
| 设备缺失 | 站点接入但部分设备未监控 | 设备类型+数量比对 |
| 信号缺失 | 设备接入但信号未配置/未上报 | 信号完整性校验 |
| 数据缺失 | 信号配置但数据长时间未更新 | 时间戳检查 |

#### 5.5.3 整改闭环

- 自动识别缺失原因（未接入/故障/网络中断/配置遗漏/已拆除未下线）
- 自动派发到对应地市/区县负责人
- 定期生成缺失率报表（按地市、设备类型）
- 缺失率趋势追踪，目标驱动至零缺失

### 5.6 动环覆盖率分析（Auditor - 覆盖率分析Agent）

> **调度模式**：定时（每日）
> **核心定位**：综资与动环数据比对，目标同比提升10%+

#### 5.6.1 三维覆盖率

```
覆盖率 = 已接入数 / 综资总数

· 站点覆盖率 = 已接入站点 / 综资站点总数
· 设备覆盖率 = 已接入设备 / 综资设备总数
· 信号覆盖率 = 已配置信号 / 标准信号总数

综合覆盖率 = 加权计算
```

#### 5.6.2 分析维度

| 维度 | 输出 |
|------|------|
| 地域 | 省→地市→区县覆盖率排名 |
| 设备类型 | 电源/空调/环境/安防各类覆盖率 |
| 时间趋势 | 月度/季度同比环比 |
| 差距分析 | 目标 vs 实际，差距清单 |
| 预测 | 基于趋势预测达标时间 |

#### 5.6.3 闭环

分析差距 → 生成缺失清单 → 监控缺失Agent跟踪 → 按地市分解目标 → 月度报告 → 达标验收

### 5.7 动环可用度分析（Auditor - 可用度分析Agent）

> **调度模式**：定时（每日）
> **核心定位**：多维评估 + 主动巡检，目标同比提升10%+

#### 5.7.1 五维评估模型

| 维度 | 权重 | 评分方法 |
|------|------|---------|
| 数据完整性 | 25% | 信号数据缺失率，数据断点统计 |
| 信号值范围 | 20% | 信号值在合理范围内的比例 |
| 告警有效性 | 20% | 有效告警占比，排除误告警/跳变 |
| 在线时长 | 20% | 月度在线率 |
| 数据存储率 | 15% | 数据成功存储占应存总量比例 |

综合可用度 = Σ(维度评分 × 权重)

#### 5.7.2 主动巡检（创新能力）

除被动数据分析外，Agent还具备**主动探测**能力：

| 探测类型 | 方式 | 目的 |
|---------|------|------|
| 命令下发探测 | 模拟下发控制指令到设备 | 验证设备是否真实响应 |
| 数据回传探测 | 请求设备主动上报一次数据 | 验证通信链路是否通畅 |
| 对比验证 | 对比动环数据与巡检系统实测数据 | 验证数据准确性 |

通过主动巡检计算**真实可用度**，与基于被动数据的**名义可用度**对比，揭示"看似正常实则异常"的设备。

| 综资系统 | 站点/设备台账 | API/数据库 |

### 5.8 告警屏蔽分析（Auditor - 屏蔽分析Agent）

> **调度模式**：定时（每日）+ 实时监控
> **核心定位**：合规审计 + 智能屏蔽策略推荐，非法屏蔽减少80%+

#### 5.8.1 合规规范

**合法场景**：
- 计划性维护（有审批工单+起止时间）
- 设备割接（有方案+审批）
- 已知误告警（经确认+备案）
- 临时调测（有申请，≤24小时）

**非法特征**（6类）：
- 无关联工单/审批
- 超7天未解除
- 范围过大（整站/整类）
- 关键告警被屏蔽（停电/高温/水浸）
- 非工作时间批量操作
- 同一人员频繁操作

#### 5.8.2 从审计到治理

传统做法仅发现违规，本方案升级为"审计+治理"：

| 能力 | 说明 |
|------|------|
| **违规检测** | 实时扫描屏蔽规则，发现违规立即触发工单（如：永久屏蔽核心网电源告警） |
| **策略推荐** | 根据历史告警频次分析，主动推荐合理的屏蔽窗口和条件，替代人工随意设置 |
| **动态调整** | 告警模式变化时（如设备更换后告警消失），自动建议解除不再需要的屏蔽规则 |

#### 5.8.3 报表

- 屏蔽总览（生效规则数、分布）
- 合规率（合法 vs 非法占比）
- 违规排行（按地市/操作人/站点）
- 月度趋势对比
- 关键告警保障率
- 整改完成率

## 12. 微服务与部署

### 12.1 微服务划分

```
网关层：  gateway-service

业务服务（8个Agent对应8个微服务）：
  anomaly-detection-service      异常监测
  ai-diagnosis-service           AI诊断
  anomaly-tracking-service       异常跟踪
  emergency-response-service     应急响应
  monitor-gap-service            监控缺失
  coverage-analysis-service      覆盖率分析
  availability-analysis-service  可用度分析
  shielding-analysis-service     屏蔽分析

能力服务：
  knowledge-graph-service        知识图谱
  rule-engine-service            规则引擎
  ai-model-service               AI模型推理
  workorder-service              工单
  notification-service           消息通知
  agent-orchestrator-service     智能体编排

基础服务：
  data-integration-service       数据集成
  auth-service                   认证授权
  system-management-service      系统管理
```

### 12.2 部署拓扑（K8s）

```
接入层：   Nginx(负载均衡+SSL) + CDN
应用层：   K8s集群
  业务Pod：  各Agent服务 × 2~3副本
  能力Pod：  知识图谱/规则引擎/Agent编排 × 2副本
  AI Pod：   AI模型推理(GPU节点) × 2副本
  计算Pod：  Flink TaskManager × N(弹性伸缩)
中间件层：  Kafka(3) + Redis(3主3从) + Nacos(3) + XXL-Job(2)
数据层：   MySQL(主从) + 时序库(3) + 图数据库(主从) + ES(3) + 向量库
```

## 13. 演进路线

### 阶段一：底座建设 + 感知能力

**范围**：
- 数据接入层搭建（动环/综资/告警数据对接）
- 动环资产知识图谱梳理
- 异常监测Agent（规则检测+统计检测+信号伪装检测）
- 监控缺失Agent（正向+反向）
- 覆盖率分析Agent
- 告警屏蔽审计Agent（合规检测部分）
- 统一预警中心（基础版大屏）

**交付物**：
- 实时异常检测能力（含信号伪装识别）
- 监控缺失自动识别+反向僵尸清理
- 覆盖率分析报表
- 屏蔽合规检测
- 统一预警展示大屏

---

### 阶段二：认知升级 + 闭环能力

**范围**：
- 知识图谱完善（故障知识+运维知识 → 动态知识图谱）
- AI诊断Agent（知识图谱推理+LLM辅助+诊断报告）
- 异常跟踪Agent（全生命周期+SLA+NLP过程监控）
- 可用度分析Agent（5维评估+主动巡检）
- 屏蔽分析Agent升级（策略推荐+动态调整）
- RAG知识问答

**交付物**：
- AI辅助诊断报告
- 异常全流程闭环跟踪
- 可用度多维分析（含主动巡检）
- 智能屏蔽策略推荐
- 运维知识问答助手

---

### 阶段三：智能运维进阶

**范围**：
- AI检测模型上线（LSTM异常检测、多信号关联检测）

- 智能体编排引擎（Agent间协同）

**交付物**：
- AI驱动的高级异常检测

**范围**：
